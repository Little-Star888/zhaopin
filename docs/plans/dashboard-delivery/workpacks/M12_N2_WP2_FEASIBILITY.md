## 1. 技术方案对比评估

### 方案 A：页面 JS 拦截（Content Script Hook）

**原理**：通过 Content Script（注入 MAIN world）Hook `XMLHttpRequest.prototype.setRequestHeader` 和 `window.fetch`，拦截智联页面发出的 API 请求，提取 `x-anit-forge-code/token` 请求头和动态 Cookie。

**技术实现路径**：

```
用户访问 sou.zhaopin.com
    |
    v
Content Script (MAIN world) 注入
    |
    v
Hook window.fetch + XMLHttpRequest.prototype
    |
    v
拦截 fe-api.zhaopin.com 请求
    |
    v
提取 x-anit-forge-code / x-anit-forge-token + Cookie
    |
    v
通过 window.postMessage -> ISOLATED world -> chrome.runtime.sendMessage
    |
    v
Background Service Worker 存储到 chrome.storage.local
```

| 维度 | 评估 |
|------|------|
| **优点** | 1. 无需逆向 JS 加密算法，参数由页面原生生成<br>2. 真实浏览器环境，指纹检测自然失效<br>3. 验证码由用户手动完成，零自动化风险<br>4. 参数自动更新（用户每次翻页都会刷新） |
| **缺点** | 1. 需要用户先访问一次智联搜索页才能初始化参数<br>2. 参数有时效性（Session 级），过期需重新访问<br>3. MV3 中需注入 MAIN world 才能读取页面请求体<br>4. 无法离线批量采集 |
| **开发复杂度** | 中等 -- 约 1-2 天 |
| **稳定性** | 中等偏高 -- 依赖页面 JS 结构不变 |
| **评分** | **8/10** |

### 方案 B：DOM 解析（类 51job 模式）

**原理**：直接解析智联招聘搜索结果页面的 DOM 结构，提取职位卡片中的文本数据，与 51job content script 实现模式一致。

**技术实现路径**：

```
用户在 sou.zhaopin.com 搜索页
    |
    v
Content Script (ISOLATED world) 注入
    |
    v
document.querySelectorAll('.joblist-box__item')
    |
    v
解析每个卡片的职位名、公司、薪资、地点等
    |
    v
归一化 -> 发送到 Background -> 存入 IndexedDB
```

| 维度 | 评估 |
|------|------|
| **优点** | 1. 实现最简单，与 51job 方案高度复用<br>2. 不涉及任何反爬参数，无逆向成本<br>3. 代码量少，维护成本低<br>4. MV3 兼容性好，无需 MAIN world 注入 |
| **缺点** | 1. 数据获取不完整 -- 只能获取当前页渲染的数据<br>2. 无法后台静默采集，需用户逐页浏览<br>3. DOM 结构变化会导致解析失败<br>4. 部分字段（如公司规模、行业）可能不在列表页 DOM 中 |
| **开发复杂度** | 低 -- 约 0.5-1 天（复用 51job 代码框架） |
| **稳定性** | 中等 -- 完全依赖 DOM 选择器 |
| **评分** | **7/10** |

### 方案 C：Cookie + API 直接调用

**原理**：通过 `chrome.cookies` API 获取智联招聘的完整 Cookie，结合从方案 A 拦截到的 `x-anit-forge-code/token`，在 Background Service Worker 中直接向 `fe-api.zhaopin.com` 发起 API 请求。

**技术实现路径**：

```
Background Service Worker
    |
    v
chrome.cookies.getAll({ domain: '.zhaopin.com' })
    |
    v
从 chrome.storage.local 读取 x-anit-forge-code/token
    |
    v
构造 fetch('https://fe-api.zhaopin.com/c/i/sou?jl=530&kw=前端&p=1')
    |
    v
携带 Cookie + x-anit-forge-* 请求头
    |
    v
解析 JSON 响应 -> 存入 IndexedDB
```

| 维度 | 评估 |
|------|------|
| **优点** | 1. 可后台静默采集，用户无需停留在搜索页<br>2. 数据完整 -- JSON API 返回结构化数据<br>3. 采集效率高，可批量请求多页<br>4. 代码量少，逻辑清晰 |
| **缺点** | 1. **强依赖方案 A 提供参数** -- 无法独立工作<br>2. Cookie 有时效性（Session 级），过期需重新登录<br>3. 频率控制需自行实现（3-8 秒/请求）<br>4. API 响应可能存在 AES 加密（需实际验证）<br>5. MV3 Service Worker 生命周期限制（可能被休眠） |
| **开发复杂度** | 中等 -- 约 1-2 天（含频率控制和错误处理） |
| **稳定性** | 中等偏低 -- Cookie 过期 + 参数时效 + 加密不确定 |
| **评分** | **6.5/10** |

### 方案 D：混合方案（A + B + C）

**原理**：方案 A（拦截）获取反爬参数，方案 C（API 调用）执行批量采集，方案 B（DOM 解析）作为兜底备选。

**技术实现路径**：

```
阶段一：参数采集（用户正常浏览时自动完成）
    |
    v
Content Script Hook -> 拦截 x-anit-forge-* + Cookie
chrome.cookies API -> 获取完整 Cookie
chrome.storage -> 持久化存储参数
    |
    v
阶段二：数据采集（后台静默执行）
    |
    v
Background Script -> 读取存储参数 -> 构造 fetch 请求
    |
    v
API 响应正常 -> 解析 JSON -> 存入 IndexedDB
    |
    v
API 响应异常（加密/参数失效）-> 降级到方案 B
    |
    v
方案 B：DOM 解析当前页 -> 存入 IndexedDB
    |
    v
阶段三：状态监控
    |
    v
Cookie 过期检测 -> 提示用户重新登录
参数失效检测 -> 提示用户重新访问搜索页
频率控制 -> 3-8秒/请求 + 随机抖动
验证码检测 -> 暂停采集 + 提示用户手动完成
```

| 维度 | 评估 |
|------|------|
| **优点** | 1. 稳定性最高 -- 多层降级保障<br>2. 数据完整性最佳 -- API 优先 + DOM 兜底<br>3. 用户体验好 -- 正常浏览即可自动采集<br>4. 鲁棒性强 -- 单点失效不影响整体 |
| **缺点** | 1. 开发成本最高 -- 需实现三套逻辑<br>2. 代码复杂度高 -- 状态管理、降级切换<br>3. 调试难度大 -- 多层通信链路 |
| **开发复杂度** | 高 -- 约 3-4 天 |
| **稳定性** | 高 -- 多重降级保障 |
| **评分** | **9/10** |

### 方案评分汇总

| 方案 | 开发成本 | 稳定性 | 数据完整性 | 用户体验 | 综合评分 |
|------|----------|--------|------------|----------|----------|
| A: 页面 JS 拦截 | 中 | 中高 | 中（仅拦截） | 良 | 8.0 |
| B: DOM 解析 | 低 | 中 | 低（仅当前页） | 一般 | 7.0 |
| C: Cookie + API | 中 | 中低 | 高（JSON） | 优 | 6.5 |
| D: 混合方案 | 高 | 高 | 高 | 优 | **9.0** |

## 3. 风险分析

### 3.1 技术风险

| 风险项 | 等级 | 概率 | 影响 | 应对措施 |
|--------|------|------|------|----------|
| MV3 MAIN world 注入被限制 | 中 | 低 | 高 | 使用 `chrome.scripting.executeScript({ world: 'MAIN' })` 已在 MV3 中官方支持 |
| `chrome.webRequest` 无法读取请求体 | 中 | 确定 | 中 | 改用 MAIN world Hook fetch/XHR 读取请求体 |
| API 响应数据 AES 加密 | 中 | 中 | 高 | DOM 解析作为兜底方案；加密时降级到方案 B |
| Service Worker 休眠导致采集中断 | 低 | 中 | 中 | 使用 `chrome.alarms` 保活；记录采集进度支持断点续传 |
| `x-anit-forge-code/token` 实际非必需 | 低 | 中 | 正面 | 根据搜索结果，这两个参数在某些场景下可有可无，简化实现 |
| `x-anit-forge-code/token` 变为必需 | 中 | 中 | 中 | 方案 A 的 Hook 机制可自动适配 |

### 3.2 法律合规风险

| 风险项 | 等级 | 详情 | 应对措施 |
|--------|------|------|----------|
| **违反 robots.txt** | 中 | 智联招聘对爬虫有明确限制；违反 robots.txt 在司法实践中会被作为考量因素 | 检查 robots.txt 内容；仅采集公开职位信息 |
| **不正当竞争风险** | 高 | 智联招聘已有诉讼先例（"智联招聘诉上海逸..."案），法院认定未经授权抓取数据构成不正当竞争 | 1. 仅采集公开职位信息，不抓取简历/个人信息<br>2. 控制请求频率，不影响平台正常运营<br>3. 数据仅用于个人求职辅助，不商业分发 |
| **个人信息保护法** | 高 | 爬取用户简历等个人敏感数据可能触犯《个人信息保护法》 | **严格不采集任何个人信息**（简历、联系方式等） |
| **网络安全法** | 中 | 高频请求可能被认定为影响网络正常运行 | 控制频率在 3-8 秒/请求，远低于触发阈值 |
| **平台服务条款** | 中 | 智联招聘服务条款可能明确禁止自动化数据采集 | 用户手动触发采集，模拟正常浏览行为 |

> **重要提示**：根据搜索结果，智联招聘 2025 年累计拦截违规企业 23 万家，平台对数据安全和合规性的重视程度持续提升。建议在实施前：
> 1. 咨询专业法律意见
> 2. 明确数据使用边界（仅个人求职辅助）
> 3. 设计合理的频率控制机制

### 3.3 维护风险

| 风险项 | 等级 | 概率 | 应对措施 |
|--------|------|------|----------|
| 智联招聘页面改版导致 DOM 选择器失效 | 中 | 中高 | 设计多层级选择器兜底（参考 51job 的备用选择器模式）；监控解析失败率 |

## 5. 最终可行性结论

### 5.1 综合评估

| 评估维度 | 结论 | 说明 |
|----------|------|------|
| **技术可行性** | **可行** | Chrome Extension 环境天然绕过 JS 加密和指纹检测；MV3 API 支持所需功能 |
| **开发成本** | 中等 | 方案 D 预估 5 天，但 80% 代码可复用 51job 框架 |
| **维护成本** | 中等偏低 | 多层降级设计降低单点失效风险；月均维护 0.5-1 天 |
| **数据完整性** | 高 | JSON API 提供完整结构化数据；DOM 兜底确保不中断 |
| **用户体验** | 良好 | 用户正常浏览即可自动采集；验证码由用户手动处理 |
| **法律合规** | **有条件可行** | 仅限公开职位信息、个人使用、控制频率 |

### 5.2 可行性结论

**结论：有条件可行**

**必要条件**：

| 编号 | 条件 | 必要性 | 验证方法 |
|------|------|--------|----------|
| C1 | 仅采集公开职位信息（职位名称、公司、薪资、地点、要求），**不采集任何个人信息**（简历、联系方式、用户画像） | **强制** | 代码审查 + 数据字段白名单 |
| C2 | 请求频率控制在 3-8 秒/次，单 IP 每小时不超过 50 次 | **强制** | 内置频率控制 + 请求计数器 |
| C3 | 数据仅用于个人求职辅助，不进行商业分发或再销售 | **强制** | 产品设计层面限制 |
| C4 | 用户主动触发采集，非后台自动大规模爬取 | **强制** | UI 交互设计：用户点击"开始采集"按钮 |
| C5 | API 响应数据加密情况需实际验证 | **建议** | 开发阶段抓包验证 `fe-api.zhaopin.com` 响应是否加密 |
| C6 | `x-anit-forge-code/token` 参数必要性需实际验证 | **建议** | 测试不传这些参数时 API 是否正常返回 |

### 5.3 如不满足条件

如果条件 C5 验证结果显示 API 响应全面加密（且无法在浏览器环境中解密），则退化为**方案 B（纯 DOM 解析）**，可行性降为"基本可行"，数据完整性降低但仍然可用。

## 6. 推荐实施路线

### 6.1 推荐方案：方案 D（混合方案）的精简版

考虑到开发成本和收益的平衡，推荐先实施 **精简版混合方案**（方案 A + B 组合，暂不实现方案 C）：

**精简版架构**：

```
第一层（主）：方案 A - 页面 JS 拦截
    -> Hook fetch/XHR，拦截 API 请求和响应
    -> 直接从响应体中提取职位数据（JSON）
    -> 如果响应正常 -> 直接获得完整数据

第二层（备）：方案 B - DOM 解析
    -> 如果方案 A 拦截失败（加密/异常）
    -> 降级解析当前页面 DOM
    -> 获得当前页的职位数据
```

**精简版 vs 完整方案 D**：

| 维度 | 精简版 (A+B) | 完整版 (A+B+C) |
|------|-------------|---------------|
| 开发工时 | 2-3 天 | 5 天 |
| 后台静默采集 | 不支持 | 支持 |
| 多页批量采集 | 不支持 | 支持 |
| 数据完整性 | 中高 | 高 |
| 维护成本 | 低 | 中 |
| **推荐度** | **首选** | 后续迭代 |

### 6.2 实施步骤

```
Step 1: 实际验证（0.5 天）
    -> 访问 sou.zhaopin.com，抓包验证 API 响应是否加密
    -> 验证 x-anit-forge-code/token 是否必需
    -> 验证 Cookie 更新频率

Step 2: 方案 B 先行（1 天）
    -> 复用 51job content script 框架
    -> 实现智联搜索页 DOM 解析
    -> 实现归一化 + UnifiedJobModel 映射
    -> 端到端测试验证

Step 3: 方案 A 增强（1-2 天）
    -> 实现 MAIN world 脚本注入
    -> Hook fetch/XHR 拦截 fe-api.zhaopin.com 请求
    -> 从响应体中提取 JSON 数据
    -> 实现参数存储 + Cookie 管理
    -> 端到端测试验证

Step 4: 集成测试（0.5 天）
    -> 测试降级切换逻辑
    -> 测试频率控制
    -> 测试异常场景（Cookie 过期、验证码触发）
    -> 与现有 Dashboard 集成测试
```
