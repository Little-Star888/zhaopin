# M11 / M12 手动采集与工作台交互修复里程碑

> 执行时间：2026-03-27
> 文档用途：下一阶段实现里程碑
> 适用范围：Dashboard 手动采集、Boss 原链接、首页结果视图、工作台 50/50 分屏、收藏交互、停止/自动采集控制

---

## 1. 里程碑目标

本阶段目标如下：

1. 让 Dashboard 手动采集在联调时可控、可停、可验证
2. 让首页默认聚焦“本次采集结果”，不再被旧数据干扰
3. 修复工作台 50/50 分屏的挂载位置和滚动问题
4. 将“待投递”统一重构为“收藏”模型
5. 修复 Boss 原链接与描述统计的展示问题

---

## 2. 问题、目标与解决方案

### 2.1 停止采集按钮无效

**目标**

- 点击“停止采集”后，当前采集任务应尽快中断

**问题**

- 当前 `STOP_CRAWL` 只修改 `isRunning` 和 `crawlState.status`
- 已开始的分页、详情抓取、等待延迟不会真正停止

**解决方案**

- 方案：在采集主流程中加入”停止标志位检查”
  - 列表分页循环前后检查
  - 详情抓取循环前后检查
  - 每次 `sleep` 前后检查

### 2.2 扩展刷新后自动开始采集

**目标**

- 联调阶段刷新扩展后，不应自动拉起采集

**问题**

- 当前 `controller_only` 模式下，扩展启动会自动注册 `crawl_bootstrap`
- alarm 到时后自动从 Controller 拉任务执行

**解决方案**

- 增加配置开关：`ENABLE_AUTO_BOOTSTRAP`
- 联调/排障阶段设为 `false`
- 生产/自动调度阶段设为 `true`

### 2.3 Boss 原链接缺失

**目标**

- 岗位详情页应尽可能显示可访问的 Boss 原链接

**问题**

- 当前部分入库记录 `url = null`
- 前端只能显示“暂无原链接”

**解决方案**

- 最终方案：系统认定的 Boss 可访问原链接 = 基于 `encryptJobId` + `securityId` 构造出的详情页 URL
- 构造格式：`https://www.zhipin.com/job_detail/{encryptJobId}.html?securityId={securityId}`
- 如果字段缺失，url 设为 `null`，前端显示"暂无原链接"

### 2.4 “含描述条数”统计不准确

**目标**

- UI 展示的“含描述条数”应与实际详情可见结果一致

**问题**

- 当前 UI 可能显示：`入库 3 条，含描述 1 条`
- 但实际岗位详情中 3 条都有描述
- 代码中存在两条统计路径：
  - `allJobs.filter(j => j.description?.length > 0).length`
  - `runStats.detailDescriptionNonEmptyCount`
- **当前推测**：Dashboard 可能走的是后者分支导致值不准确（待日志验证确认）

**解决方案**

- 先做数据验证，对比两条路径的真实返回值差异
- 根据验证结果统一为正确的统计路径
- 在根因未验证前，不把任何一种解释写死

### 2.5 首页旧数据残留

**目标**

- 每次新搜索开始时，首页只展示当前这轮采集结果

**问题**

- 当前首页是 SQL 全量库视图
- 新一轮搜索开始后，旧产品经理等历史数据仍会继续显示

**解决方案**

- 首页切换为“本次采集结果视图”
- 每次手动采集开始时，进入当前批次模式
- 首页只展示当前 `crawl_batch_id` 对应的数据
- 保留“全部岗位”入口，允许用户手动切回全库视图
- 不删除历史数据

### 2.6 工作台 50/50 分屏位置错误

**目标**

- 点击工作台岗位卡片后，50/50 分屏只在工作台页面内展示

**问题**

- 当前分屏会跑到第二页
- 首页也会出现分屏内容
- 用户需要滚动页面才能看到

**解决方案**

- `#home` 只负责首页卡片流
- `#resume` / 工作台页负责 7:3 与 50/50 的切换
- `splitView` 只挂载在工作台容器内部
- 打开岗位卡片时：
  - 隐藏 7:3 工作台
  - 直接显示 50/50 分屏
- 关闭分屏时：
  - 恢复 7:3 工作台
- 分屏不参与正常文档流，不再导致页面被撑长

### 2.7 “待投递”语义错误，应统一为“收藏”

**目标**

- 统一交互语义、按钮文案和状态视觉

**问题**

- 当前系统没有真实“投递”动作
- “加入待投递 / 已投递 / 取消选择”与实际功能不匹配

**解决方案**

- 将交互统一为“收藏模型”
- 首页：
  - 未收藏：黄色五角星 + `收藏`
  - 已收藏：灰色五角星
- 工作台详情：
  - `取消收藏`
- 取消收藏后联动更新：
  - 关闭当前 50/50 分屏
  - 从工作台列表移除该岗位
  - 首页卡片五角星恢复未收藏状态

### 2.8 首页刷新闪动

**目标**

- 首页刷新时保持稳定，不打断用户浏览

**问题**

- 之前轮询刷新会整页重绘，造成明显闪动

**解决方案**

- 保留当前已实施的静默刷新方案
- 数据不变时不重渲染
- 后续批次视图接入后继续沿用

### 2.9 手动与自动模式需要明确区分

**目标**

- 保留双模式能力，同时避免后续逻辑继续混杂

**问题**

- 当前排障阶段已经加入手动绕过冷却、手动关键词优先等特殊逻辑
- 若不拆分，会污染自动模式行为

**解决方案**

- 后续在消息协议和运行时配置中显式区分：
  - 手动模式：用户输入关键词 + 人工触发
  - 自动模式：Controller 队列 + 预设规则
- 冷却、过滤、采集预算按模式拆分配置

### 2.10 手动冷却策略需要收敛

**目标**

- 手动模式可测试，但不能完全绕过冷却

**问题**

- 当前排障实现允许手动模式绕过 blocked window

**解决方案**

- 排障阶段保留当前行为
- 架构收敛阶段改为短冷却窗口
- 推荐值：`3-5 分钟`

## 4. 本阶段需要验证的事项

以下事项在实现时必须做数据验证：

1. 构造 URL 是否在真实环境中稳定可访问（最终方案，不再保留 DOM 提取回退）
2. `allJobs.filter(...)` 与 `detailDescriptionNonEmptyCount` 的差异
3. `crawl_batch_id` 过滤后，首页是否只显示本轮数据
4. 50/50 分屏是否完全从首页剥离
5. 收藏 / 取消收藏后，首页、工作台、分屏三处状态是否一致

---

## 5. 本阶段输出标准

本里程碑完成时，应满足以下标准：

1. 手动联调时，扩展刷新后不会自动开始采集
2. 点击“停止采集”后，当前采集任务可停止
3. 首页默认展示当前批次结果，不再残留旧搜索结果
4. Boss 详情可显示可访问的原链接或明确的无链接状态
5. “含描述条数”与实际详情一致
6. 工作台点击卡片后，50/50 分屏只在工作台内出现
7. “待投递”相关文案全部替换为“收藏”模型

## 7. 后续优化（不在 M13 范围）

以下问题由代码审查发现，属于不同子系统或未来优化方向，不纳入 M13，建议在后续里程碑中处理：

### 7.1 delivery_queue 全表内存溢出风险
- **位置**：`controller/db.js` → `updateDeliveryPayloadCompanyFieldsByNormalizedName`
- **问题**：`SELECT id, payload... FROM delivery_queue` 加载全表到内存遍历解析 JSON
- **风险**：数据量达到几千条时可能阻塞事件循环或 OOM
- **建议**：改为 LIMIT/OFFSET 分页处理，或在 SQL 层做 JSON 过滤

### 7.2 delivery_queue 查询索引失效
- **位置**：`controller/db.js` → `getDeliveryStats` 等查询
- **问题**：`ORDER BY datetime(created_at)` 对列使用函数导致索引失效，退化为全表扫描
- **建议**：ISO8601 字符串支持字典序，直接 `ORDER BY created_at` 即可命中索引

### 7.3 ai_configs 表缺少 UNIQUE 约束
- **位置**：`controller/db.js` → `migrateToV8`
- **问题**：`INSERT OR IGNORE INTO ai_configs` 但 `provider` 字段无 UNIQUE 约束，IGNORE 不生效
- **建议**：`CREATE TABLE` 中为 `provider TEXT NOT NULL UNIQUE`

### 7.4 content.js fetch 未绑定 AbortController
- **位置**：`crawler/content.js` → `executeInPageContext`
- **问题**：30 秒超时 reject Promise 后，底层 fetch 仍在 pending
- **风险**：频繁超时时可能耗尽浏览器并发连接池
- **建议**：引入 AbortController，超时触发时 abort 底层请求
