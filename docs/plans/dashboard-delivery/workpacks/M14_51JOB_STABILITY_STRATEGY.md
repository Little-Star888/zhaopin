# 智联招聘 / 前程无忧稳定爬取方案

## 目标

在不依赖绕过风控的前提下，建立一套相对稳定、可长期维护、尽量不漏爬的招聘采集方案。

目标拆分如下：

1. 智联招聘：
   - 稳定抓取列表页岗位
   - 在可控强度下补抓部分详情页
   - 将详情抓取失败降级为非阻塞，而不是整批任务失败

2. 前程无忧：
   - 稳定抓取列表页岗位
   - 详情页在当前站点风控约束下不作为主链路成功条件
   - 对详情不可抓状态进行显式标记

3. 两个平台：
   - 统一入库 schema
   - 明确区分列表成功、详情成功、详情失败、风控拦截
   - 尽量通过多轮补抓减少漏爬

---

## 背景

当前扩展最初主要围绕 Boss 直聘实现，后续补了智联招聘和前程无忧，但这两条分支存在明显的“旧页面结构 + 固定等待 + 旧 URL 假设”的问题。

在本轮排查中，已经做了以下事实验证：

### 1. 智联招聘并不是“完全不能自动抓”

实际探针结果：

- 南京“产品”列表页成功，第一页可见约 20 条岗位
- 上海“产品”列表页成功，第一页可见约 20 条岗位
- 同一浏览器会话内，连续访问 4 个南京详情页，都拿到了正文描述

说明：

- 智联当前可以稳定抓列表页
- 在低频、少量详情访问时，详情页也可以成功
- 问题不在于“自动访问必然失败”，而在于采集强度、页面结构兼容和错误判定方式

### 2. 前程无忧详情页在当前环境下会进入访问验证

实际探针结果：

- 北京、上海列表页都能成功抓取
- 同一会话中打开 2 个详情页，均进入“滑动验证页面”

说明：

- 前程无忧列表页可以作为稳定主链路
- 详情页自动补抓不适合作为稳定方案的核心组成部分
- 任何把 51job 详情抓取当作必达目标的方案，都会不稳定

### 3. 当前代码中的实现问题会放大误判

包括但不限于：

- 智联 / 51job 分支最初无视 dashboard 传入的 `city` 和 `keyword`
- 智联仍在使用旧的搜索 URL 与详情 URL 假设
- 智联 `jobId` 解析不兼容 `CCL...J...` 这类新格式
- 51job 仍按旧站 `search.51job.com/list/...html` 结构解析，而当前主站已是 `we.51job.com/pc/search`
- 原实现把“未加载完成 / 登录态 / 验证页 / 空结果 / 结构变更”统一报成 `NO_DATA`
- 智联错误识别中过度依赖页面文本里的“登录/验证”等词，容易误判

**结论：完全同意”从单字段升级为最小状态机”的方案。** 这是当前方案从”战术修补”走向”系统能力”的关键转折点。

##### 状态机设计的具体建议

基于用户提出的 4 个补充字段，顾问建议完整的字段清单如下：

```sql
-- 详情采集状态机字段组
detail_status        TEXT DEFAULT 'pending'  -- pending/success/skipped/anti_bot/empty/error
detail_attempt_count INTEGER DEFAULT 0       -- 累计尝试次数
last_detail_attempt_at DATETIME              -- 上次尝试时间
next_detail_retry_at   DATETIME              -- 下次可重试时间（退避策略）
detail_error_code      TEXT                   -- 具体错误码（login_required/timeout/parse_fail/structure_drift）
```

##### 为什么用户提出的 3 个痛点是准确的

1. **`anti_bot` 无法知道何时适合重试** — 同意。没有 `next_detail_retry_at`，调度器只能每次全量扫描 `pending`，无法实现指数退避。
2. **`empty` 无法区分页面无正文 vs 解析失败** — 同意。这是最危险的静默错误，会导致有效数据被标记为”已处理”而永远不会重试。`detail_error_code` 字段可以精确区分这两种情况。
3. **`pending` 每轮被重复扫描** — 同意。没有 `last_detail_attempt_at` 和 `next_detail_retry_at`，调度效率极低。应该只扫描 `next_detail_retry_at <= NOW()` 的记录。

##### 建议的退避策略

```
首次失败 → next_detail_retry_at = NOW() + 5min
第二次失败 → NOW() + 30min
第三次失败 → NOW() + 2h
第四次+ → NOW() + 24h
达到 max_attempts (如 5) → detail_status = 'skipped'，不再重试
```

##### 额外建议

用户提出的字段组已经足够覆盖当前需求。如果后续要支持”手动触发补抓”，可以考虑增加一个 `manual_retry_requested BOOLEAN DEFAULT 0` 字段，允许用户在 Dashboard 中手动将 `skipped` 状态的岗位重新标记为可重试。

## 原因分析

## 智联招聘能成功的原因

根据实际探针，智联成功的核心条件是：

1. 列表页是公开可见内容
2. 单次访问频率不高
3. 同一会话内只抓少量详情
4. 页面结构已经对齐现网 DOM

因此，智联的稳定性更依赖：

- 访问强度控制
- DOM 兼容
- 错误分类与自动降级

而不是依赖任何绕过风控动作

#### 顾问补充意见

> 来源：与顾问讨论确认

**顾问认同此分析**，同时补充：

1. **智联的"稳定"是动态平衡**：当前低频可抓不代表长期可行，智联的风控策略可能随时收紧。建议在代码中增加"连续失败计数器"，当连续失败超过阈值时自动停止本轮采集并记录状态，而不是继续重试。

2. **DOM 兼容是隐性风险**：智联和 51job 的前端架构会不定期更新。当前 content script 对 DOM 结构有较强假设（特定 class 名、特定层级关系），一旦改版会导致大面积提取失败。建议在采集结果中增加"DOM 结构签名"校验，当签名变化时触发告警而非静默返回空数据。

3. **双通道机制是好的设计**：智联的"API 拦截 + DOM 回退"双通道策略是正确的架构选择，应保持并推广到其他平台。

## 前程无忧不稳定的原因

前程无忧的问题不是列表，而是详情：

1. 列表页本身公开可见
2. 详情页在自动访问下更容易进入访问验证

## 如何尽量不漏爬

“不漏爬”不能等价于“当轮就拿全所有详情”。

更现实的工程目标应该是：

1. 列表岗位尽量全量入库
2. 详情允许延后补抓
3. 使用多轮机制逐步补全

建议做法：

### 1. 列表成功优先落库

无论详情是否成功，只要列表拿到，就先写入：

- `platform`
- `platformJobId`
- `title`
- `company`
- `location`
- `url`
- `salary`
- `experience`
- `education`
- `keywords`

### 2. 增加详情状态字段

建议增加：

- `detailStatus: success | skipped | anti_bot | login_required | empty`
- `listCapturedAt`
- `detailCapturedAt`

这样可以支持后续补抓，而不是反复重抓整个列表

### 3. 使用 platformJobId 去重

以 `platform + platformJobId` 作为主去重键，避免：

- 列表重复写入
- 详情补抓重复写入

### 4. 分批补抓详情

对智联：

- 每轮只补前 N 个尚未完成详情的岗位
- 下一轮继续补剩余岗位

对 51job：

- 默认不自动补详情
- 只有在发现某些会话条件下详情可访问时才启用补抓

### 5. 记录采集批次和失败原因

建议每轮记录：

- 城市
- 关键词
- 页面数
- 列表抓取数量
- 详情尝试数量
- 详情成功数量
- 被风控数量
- 空正文数量

这样可以持续校准“安全阈值”

#### 我的意见

这一节还缺“完整爬取能力”的核心定义。

完整能力不是一次跑完，而是系统具备最终补齐的能力。为此建议再补 5 个机制：

1. **分页队列**
   - 每个 `city + keyword` 都要有页码任务
   - 页码状态至少包括：`pending / running / done / failed`

2. **详情 backlog**
   - 列表入库后，把需要补详情的岗位放入 backlog
   - backlog 按优先级消费，而不是每轮只取“当前页前 N 条”

3. **断点恢复**
   - 插件刷新、浏览器关闭、用户停止后
   - 下次应从未完成页码或 backlog 继续

4. **页终止判定**
   - 空页终止
   - 重复率过高终止
   - 连续两页无新岗位终止

5. **跨页/跨轮去重**
   - 不只 `platformJobId`
   - 还要考虑刷新上架、重新发布、跨天更新等情况

如果不补这些内容，这份方案更像”稳定抓样本”的方案，而不是”完整爬取能力”的方案。

#### Claude + 顾问讨论意见

> 2026-03-28 | 顾问联合分析

**结论：完全同意。** 当前方案本质上是”稳定采样”而非”完整采集”。用户提出的 5 个机制是将系统从采样器升级为采集引擎的必要条件。

##### 对 5 个机制的优先级排序建议

不是 5 个机制都需要立即实现。按”最小可用”原则排序：

| 优先级 | 机制 | 理由 | 预计工期 |
|--------|------|------|---------|
| **P0** | 跨页/跨轮去重 | 不实现等于每次都全量重查，浪费请求且加重风控 | 0.5 天 |
| **P0** | 断点恢复 | 插件刷新/浏览器关闭后数据丢失是最常见的失败场景 | 1 天 |
| **P1** | 分页队列 | 没有 p2+ 就永远只有每城市 20-30 条 | 1 天 |
| **P2** | 详情 backlog | 依赖 detailStatus 状态机先落地 | 1.5 天 |
| **P2** | 页终止判定 | 连续空页/高重复率终止可避免无效请求 | 0.5 天 |

##### 关于”页终止判定”的具体参数建议

```
终止条件（满足任一即停止翻页）：
1. 空页：当前页返回 0 条岗位
2. 高重复率：当前页新岗位占比 < 10%（即 90% 以上与已入库岗位重复）
3. 连续无新：连续 2 页无任何新岗位
4. 硬上限：单次任务最多翻 N 页（默认 10，可配置）
```

##### 关于”跨页/跨轮去重”的细化

用户提到”不只 platformJobId，还要考虑刷新上架、重新发布、跨天更新”，这涉及去重策略的时效性：

- **短时去重（同一批次内）**：`platform + platformJobId` 精确去重
- **跨天去重（不同批次间）**：`platform + platformJobId` 仍为主键，但需要 `updated_at` 字段记录最近一次看到该岗位的时间，用于区分”岗位仍在招”和”岗位已下架”
- **跨平台去重（不同平台间）**：`title + company` 模糊匹配，但这属于下游分析逻辑，不应在采集层实现

**结论：完全同意用户将优先级从"战术修补"调整为"能力建设"。** 这是正确的工程决策。

##### 为什么"数据与调度基础设施"必须是第一优先级

顾问的核心论点：**没有基础设施的业务代码是空中楼阁。**

具体来说：

1. **detailStatus 是"后端补抓"的前提条件**
   - 没有它 → 每次补抓只能全量扫描 → 风控压力倍增
   - 有了它 → 只扫 `detail_status IN ('pending', 'anti_bot')` → 效率提升 10x+

2. **页码任务表是"分页抓取"的前提条件**
   - 没有它 → 只能硬编码 p1 → 能力上限锁死在 20-30 条/城市/关键词
   - 有了它 → 支持 p1 到 pN 的渐进式扩展 + 断点恢复

3. **去重是"多轮叠加"的前提条件**
   - 没有它 → 每轮抓取都可能产生大量重复数据
   - 有了它 → 多轮采集可以安全叠加，数据只会增加不会膨胀

##### 关于"原始优先级"的评价

原文的优先级排序（智联链路 → 51job 固化 → 基础设施）有一个隐含问题：它假设"先跑通再补基础设施"。但实际工程经验表明，**没有基础设施的"跑通"是不可重复的**——这次跑通了，下次换个城市或关键词就又可能失败，因为缺乏状态追踪和恢复能力。

用户的优先级重排（基础设施 → 智联链路 → 51job 规模化）更符合"先打地基再盖楼"的工程原则。

##### 关于 51job 的"工程预算"判断

同意"51job 当前不适合把工程预算投入到详情自动化上"。补充一个量化依据：

- 列表页风控触发率：< 5%（极低，稳定可用）
- 详情页风控触发率：> 80%（极高，不可用）
- 详情页的 ROI 极低：投入 1 天工程时间，可能只能多抓 2-3 条详情

**ROI 排序决定预算分配**：51job 列表 > 智联列表 + 详情 > 51job 详情。

## 最终结论

### 智联招聘

可以做相对稳定的自动爬取。

稳定方式不是“全量高频自动抓详情”，而是：

- 列表优先
- 少量详情补全
- 失败自动降级

### 前程无忧

当前能稳定的是列表，不是详情。

因此稳定方案必须是：

- 列表为主链路
- 详情不是强依赖

### 对业务目标的影响

如果目标是“尽量不漏爬”，那么最正确的工程定义是：

- 不漏掉列表岗位
- 详情允许延迟补抓
- 通过多轮机制逐步补全

而不是要求”每个平台、每个岗位、每轮都拿到全文详情”。

## 统一后实施方案

在综合现有分析、代码验证、探针结果和补充意见后，建议将本方案统一收敛为下面这一版执行口径。

### 一、统一目标

本项目的目标不是“单轮一次性抓完所有详情”，而是建立一套：

- 能稳定抓到尽可能完整的列表岗位
- 能对可抓详情进行持续补全
- 能在失败后自动降级、自动恢复
- 最终具备“逐轮逼近完整”的系统能力

需要明确区分两件事：

1. **测试规模**
   - 首轮测试可以只测少量城市、关键词、详情页
   - 这是验收策略，不是能力边界

2. **最终能力**
   - 智联应支持多城市、多关键词、多页列表抓取，并通过 backlog 逐步补全详情
   - 51job 应支持多城市、多关键词、多页列表抓取，但当前不承诺详情全文自动完整补齐

### 二、统一结论

#### 智联招聘

智联可以作为“列表 + 详情补全”的主攻平台。

已验证结论：

- 列表页可以稳定抓取
- 少量详情页在低频条件下可成功拿到正文
- 适合采用“列表优先 + 详情预算补全 + 自动降级”的架构

但需要强调：

- “当前样例成功”不等于“完整能力已完成”
- 还必须补齐分页调度、详情 backlog、状态机、断点恢复等基础设施

#### 前程无忧

51job 当前只能作为“完整列表抓取”的主链路平台。

已验证结论：

- 列表页可稳定抓取
- 详情页自动访问会进入滑动验证

因此必须明确：

- 当前能承诺的是“列表完整性优先”
- 不能承诺“详情全文稳定自动补齐”

### 三、统一的工程边界

#### 可承诺的能力

1. 智联：
   - 列表尽量完整
   - 详情逐轮补全

2. 51job：
   - 列表尽量完整
   - 详情状态可追踪

#### 不应承诺的能力

1. 所有平台、所有岗位、每轮都拿到全文详情
2. 在不变更平台策略的前提下，51job 详情稳定自动补全

### 四、统一的架构方案

#### 1. 数据层先行

优先落地以下字段 / 结构：

- `detail_status`
- `detail_attempt_count`
- `last_detail_attempt_at`
- `next_detail_retry_at`
- `detail_error_code`
- 页码任务表 / 队列
- 批次日志

原因：

- 没有这些，所谓“多轮补抓”和“完整能力”都不成立

#### 2. 列表抓取和详情抓取彻底拆开

统一原则：

- 列表成功即入库
- 详情抓取是异步补全流程
- 详情失败不影响列表成功

这意味着系统层面必须允许：

- 一条岗位先以 `detail_status = pending` 入库
- 后续多轮任务再去补正文

#### 3. 智联采用两阶段模型

##### 阶段 A：接入稳定期

- 单城市单关键词串行
- 默认先抓 `p1`
- 每轮只补少量详情
- 连续失败后自动降级为列表模式

##### 阶段 B：完整回填期

- 开启分页任务
- 对 `pending` / `anti_bot` 的详情建立 backlog
- 多轮预算化补抓
- 逐步逼近详情完整

#### 4. 51job 采用列表稳定模式

- 列表多页抓取
- 详情默认 `skipped` 或 `anti_bot`
- 暂不把 51job 详情补全作为主路线

### 五、统一的分页与补抓机制

如果最终目标是“尽量不漏爬”，必须有以下机制：

#### 分页队列

每个 `platform + city + keyword + page` 都应该有任务状态：

- `pending`
- `running`
- `done`
- `failed`

#### 页终止条件

建议统一采用：

- 空页终止
- 连续两页无新岗位终止
- 重复率过高终止

#### 详情 backlog

详情补抓不应只依赖“当前页前 N 条”，而应从 backlog 中按优先级消费：

- `pending`
- `anti_bot`（到达重试时间后）
- `empty`（视业务决定是否重试）

#### 断点恢复

系统必须支持：

- 插件刷新后继续
- 用户停止后继续
- 浏览器关闭后继续

### 六、统一的优先级

#### 第一优先级：能力基础设施

- 详情状态机字段组
- 页码任务表
- 去重
- 断点恢复

#### 第二优先级：智联完整主链路

- 列表分页
- 详情预算补全
- 连续失败自动降级

#### 第三优先级：51job 列表规模化

- 列表分页
- 列表去重
- 详情状态化跳过

### 七、统一的完成标准

#### 验收阶段

为了控制风险，首轮验收可以只验证少量样本：

- 少量城市
- 少量关键词
- 少量详情页

#### 最终完成标准

真正的完成标准应是：

1. 智联
   - 可稳定跑多页列表
   - 可通过 backlog 多轮补全详情
   - 失败后可自动降级和恢复

2. 51job
   - 可稳定跑多页列表
   - 详情状态有记录
   - 不因详情失败丢失列表数据

### 八、建议采用的最终表述

如果需要把整份文档收束成一句话，建议统一口径为：

> 本方案的目标不是保证每轮抓全所有详情，而是先保证列表尽量完整，再通过状态机、分页任务和详情 backlog 建立一个可持续补齐的爬取系统；其中智联作为“列表 + 详情补全”主链路，51job 作为“列表完整性优先”主链路。
