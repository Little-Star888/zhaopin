## 疑问1：里程碑划分与执行顺序

### 背景
M7/M8/M9 文档链已完成（3 里程碑 + 8 节点 + 35 工作包）。用户提出三个新任务，需要确定如何组织为里程碑以及执行顺序。

### 目标
在最小改动范围内，合理规划三个任务的里程碑划分和执行顺序。

### 问题
三个任务之间存在依赖关系，不同的执行顺序会影响工作量：
- UI 修复可能需要为爬虫按钮预留位置
- 多平台爬虫可能需要先完成 UI 集成才能测试
- Dashboard 的 layout 变化可能影响所有后续改动

### 可选方案

| 方案 | 顺序 | 优点 | 缺点 | 风险 |
|------|------|------|------|------|
| **A. UI先行** | M10(UI修复) → M11(UI集成) → M12(爬虫) | 先把UI骨架定好，后续改动有稳定基础 | UI修复工作量大，延迟数据产出 | 中 |
| **B. 数据先行** | M12(爬虫) → M11(UI集成) → M10(UI修复) | 先有数据支撑，爬虫扩展独立于UI | UI修复可能需要重做已集成的部分 | 高 |
| **C. 并行推进** | M10(UI修复) ∥ M12(爬虫) → M11(集成) | 最快交付 | 需要协调两个方向的改动，dashboard.js 冲突 | 高 |
| **D. 增量融合** | M10(差异修复+爬虫UI预留) → M11(爬虫扩展+集成) | 减少里程碑数量，一次改动到位 | 单个里程碑过大 | 中 |

### 分歧点

| | 我的方案 | 顾问方案 |
|--|---------|---------|
| 里程碑划分 | 倾向方案D（增量融合，减少里程碑） | 方案A（UI先行，M10/M11/M12 三里程碑） |
| 理由 | M10和M12可以并行不冲突（CSS vs content.js），合并可减少协调成本 | UI结构必须先稳定，否则后续集成要在不稳定的基础上叠加 |

### 待决策项
- [ ] 选择哪个方案？（A/B/C/D）
- [ ] M10是否应包含"为爬虫按钮预留位置"的工作？

### 我的解答
**采用方案 A 的变体**：

- 保持 `M10(UI结构与设计稿对齐) → M11(采集控制集成到 Web UI) → M12(多平台爬虫扩展)` 的大顺序不变。
- 但 M12 内部必须再拆成多个轮次，不把所有平台一次性塞进同一个开发轮次。
- `M10` 应包含“为爬虫按钮预留位置”的工作，因为用户已经确认后续会把采集控制放进 Web UI。

执行解释：

- 先把 UI 骨架定住，后续按钮和状态位才不会反复挪位置。
- 爬虫平台难度差异过大，M12 不应被理解为“一次做完所有平台”，而是“启动多平台扩展主线”。

## 疑问4：导航结构（Hash路由 vs 滚动导航）

### 背景
设计稿使用基于滚动的锚点导航：
- 固定顶部 `.vnav`（黑色背景+红色底边）
- 只有"首页"和"工作台"两个 tab
- 每个 section 内有 `.mock-nav` 子导航
- 通过 `scrollY` 检测当前 section 来高亮 tab

当前实现使用 Hash 路由：
- `#home` / `#resume` / `#ai-config` 三个 tab
- 通过 `hashchange` 事件切换视图的 `display` 属性

### 目标
确定导航方式，使其与设计稿一致。

### 问题
- 滚动导航意味着首页和工作台在同一个长页面中，用户可以上下滚动
- Hash 路由是视图切换，一次只显示一个 section
- 如果改用滚动导航，整个页面结构需要重大调整
- 但 Mockup 是一个独立的 HTML 文件，所有逻辑都在一个页面内；Dashboard 是模块化的，通过 ES Module 加载

### 可选方案

| 方案 | 描述 | 改动范围 | 与设计稿一致性 |
|------|------|---------|---------------|
| **A. 改用滚动导航** | 按 mockup 的 scroll-based 方式重构 | 大改 dashboard.html 结构 + dashboard.js 路由逻辑 | 完全一致 |
| **B. 保持 Hash 路由** | 保留 #home/#resume/#ai-config，精简为两 tab | 小改（删除 ai-config tab） | 不一致 |
| **C. 视觉一致即可** | 保持 Hash 路由逻辑，但导航栏样式改为 mockup 的 .vnav 风格 | CSS + 少量 HTML | 视觉一致，交互不一致 |

### 分歧点

| | 我的方案 | 顾问方案 |
|--|---------|---------|
| 首选 | C. 视觉一致即可（最小改动） | B. 保持 Hash 路由（"保留现有的 Hash 路由逻辑，不大改JS"） |
| 理由 | 滚动导航改动太大，且 Hash 路由在模块化项目中更合理；视觉上对齐设计稿即可满足用户需求 | 避免过度重构 |

> 顾问选择了方案B（保留Hash路由但精简tab），我倾向于方案C（保留Hash路由+视觉对齐）。差异在于是否需要删除 #ai-config tab。

### 待决策项

- [ ] 导航栏是否需要改为 .vnav 样式（黑色背景+红色底边+大写字母）？
- [ ] 是否需要添加 .mock-nav 子导航？

### 我的解答

具体口径：

- 不回退成设计稿原始的两 tab。

## 疑问5：爬虫架构扩展方式（函数式 vs 类/策略模式）

### 背景
当前爬虫是纯函数式实现：
```javascript
// content.js - 直接在页面上下文中执行
window._bossScraper = {
  async scrapeJobs(keyword, cityCode, page) { /* fetch zhipin.com/wapi */ },
  async getJobDetail(encryptJobId, lid, securityId) { /* fetch detail API */ }
};
```

顾问建议引入 BaseScraper 类和策略模式：
```javascript
class BaseScraper { extract() { throw new Error('Not implemented'); } }
class BossScraper extends BaseScraper { extract() { /* Boss逻辑 */ } }
class LiepinScraper extends BaseScraper { extract() { /* 猎聘逻辑 */ } }
// hostname 策略分发
```

### 目标
以最小改动扩展多平台爬虫支持，同时保持代码可维护性。

### 问题
- 项目约束：纯 JavaScript CommonJS + JSDoc，零框架
- 引入 ES6 class 在浏览器 content script 中是可行的（现代浏览器都支持）
- 但在 Node.js 后端（controller/）中使用 class 需要确认与 CommonJS 的兼容性
- 策略模式是否"过度重构"？

### 可选方案

| 方案 | 描述 | 改动范围 | 可维护性 |
|------|------|---------|---------|
| **A. 策略模式(class)** | BaseScraper + 各平台子类 + hostname 分发 | 中等（新增抽象层，封装现有逻辑） | 高 |
| **B. 函数式分发** | 保留函数式，用对象映射分发：`const scrapers = { 'zhipin.com': bossScraper, 'liepin.com': liepinScraper }` | 小（不引入class，只加分发逻辑） | 中 |
| **C. 独立文件** | 每个平台一个独立 content script，manifest.json 中按 hostname 匹配 | 小（零重构，完全隔离） | 中（文件多但逻辑独立） |
| **D. 不重构** | 直接在 content.js 中 if/else 添加新平台逻辑 | 最小 | 低（代码膨胀） |

### 分歧点

| | 我的方案 | 顾问方案 |
|--|---------|---------|
| 首选 | C. 独立文件（manifest.json hostname匹配）或 B. 函数式分发 | A. 策略模式(class) |
| 理由 | 方案C完全不需要重构现有代码，每个平台独立文件，零耦合。方案B不引入class，保持函数式风格。符合"最小改动"约束。 | 策略模式有更好的可扩展性，添加新平台只需新增一个类 |

### 待决策项
- [ ] 选择哪种爬虫扩展方式？（A/B/C/D）
- [ ] 如果选择 class 方式，是否与项目的"纯JS CommonJS"约束冲突？

### 我的解答
**当前不拍板 class/策略模式。先按更保守的方向准备：独立文件或函数式分发优先。**

原因：

- 用户已经明确多平台难度跨度很大，需要分轮次推进。
- 在 API 可行性、验证码约束都没研究清楚之前，先做重抽象风险过高。
- 最小改动原则下，先让简单平台接入，再根据第二个平台的真实差异决定是否上更强抽象。

当前建议顺序：

1. 先做最简单的平台验证
2. 再研究高门槛平台（含手机验证码）
3. 之后再决定是否需要统一抽象层

### 问题

- 如果没有内部 API，需要使用 DOM 解析或第三方接口，架构可能完全不同
- 这个不确定性直接影响疑问5（爬虫架构）的方案选择

### 可选方案

| 场景 | 描述 | 影响 |
|------|------|------|
| **A. 都有内部API** | 各平台都有类似 wapi 的内部接口 | 统一架构，策略模式可行 |
| **B. 部分有API** | 有的有内部API，有的需要DOM解析 | 需要混合架构 |
| **C. 都没有API** | 都需要DOM解析 | 需要统一的DOM解析框架 |

### 分歧点

| | 我的方案 | 顾问方案 |
|--|---------|---------|
| 判断 | **未验证**，需要实际调研各平台的API情况 | 未明确提及，假设"DOM解析逻辑" |

> **这是一个需要实际调研的关键问题**，直接影响爬虫架构设计。

### 待决策项
- [ ] 调研猎聘(liepin.com)是否有内部 API
- [ ] 调研51job(51job.com)是否有内部 API
- [ ] 调研智联(zhaopin.com)是否有内部 API
- [ ] 如果没有内部API，是否考虑使用各平台官方开放API？

### 我的解答
**本项不直接定技术方案，先进入研究。**

用户已确认的现实约束：

- 有一个网站很简单，普通爬虫即可

因此这里的结论应写成：

- 在研究完成前，不预设统一抓取架构

## 疑问7：Dashboard 运行环境与通信方式

### 背景
爬虫采集按钮需要集成到 Web UI（Dashboard），但需要确定 Dashboard 的运行环境：

- **Chrome Extension 页面**（popup.html、options_page）：可以直接使用 `chrome.runtime.sendMessage` 与 background.js 通信
- **独立网页**（通过 localhost 访问）：只能通过 HTTP API 与 Node.js 后端通信

### 目标
确定 Dashboard 的运行环境，从而决定爬虫控制的通信方式。

### 问题
- 当前 manifest.json 的配置决定了 Dashboard 的运行方式
- 如果 Dashboard 是扩展页面，可以直接发送消息给 background.js 来触发爬虫
- 如果 Dashboard 是独立网页，需要通过 Node.js 后端间接调用爬虫
- 这决定了 M11（采集控制集成）的技术方案

### 可选方案

| 方案 | 描述 | 通信方式 | 复杂度 |
|------|------|---------|--------|
| **A. Extension页面** | Dashboard 作为 Chrome Extension 的一部分运行 | chrome.runtime.sendMessage → background.js | 低 |
| **B. 独立网页** | Dashboard 通过 localhost 访问，Node.js 提供 HTTP API | fetch('/api/crawl/start') → Node.js → chrome.runtime（需要bridge） | 高 |
| **C. 混合** | Dashboard 同时支持两种方式 | 环境检测 + 双通道 | 中 |

### 分歧点

| | 我的方案 | 顾问方案 |
|--|---------|---------|
| 判断 | 需要查看 manifest.json 确认当前配置 | 未明确分析，假设"postMessage 或 fetch API" |

### 待决策项
- [ ] 确认 Dashboard 当前的运行方式（Extension页面 vs 独立网页）
- [ ] 如果是独立网页，是否需要新增 bridge 服务连接 Extension 和 Web UI？

### 我的解答
**按当前 Extension 页面理解继续规划。**

原因：

- 用户给出的当前地址是 `chrome-extension://.../dashboard.html#resume`
- 这说明当前 Dashboard 实际运行在 Chrome Extension 页面内

因此：

- 优先按 Extension 页面通信方式设计
- 暂不引入额外 bridge 服务
- 若未来真的迁移到独立网页，再单独讨论 bridge

| | 我的方案 | 顾问方案 |
|--|---------|---------|
| 首选 | B. 只实现子导航（功能相关） | 未明确提及 |

### 待决策项
- [ ] 装饰性元素（.vlabel、.anim动画）是否需要在本次迭代中实现？

**按设计稿实现。**

但补一个执行边界：

## 疑问10：AI匹配功能的位置

### 背景
设计稿中，AI匹配功能出现在两个位置：
1. **工作台右侧简历面板**（`.ws-res`）：`AI 优化` + `AI 匹配` 按钮
2. **50/50分屏右侧**（`.sp-r`）：同样的按钮

当前实现中：
- AI 优化：在简历工具栏中（`res-btn--ai`）
- AI 匹配：在简历工具栏中

### 目标
确定 AI 匹配功能在 UI 中的位置和交互方式。

### 问题
- 设计稿中 AI 匹配是对"当前选中的岗位"进行匹配
- 当前实现中 AI 匹配似乎是对"待投递列表中的岗位"进行匹配
- 如果改用 50/50 分屏，左侧是具体岗位详情，右侧是简历，AI 匹配应该是对这个具体岗位进行匹配

### 可选方案

| 方案 | 描述 | 改动范围 |
|------|------|---------|
| **A. 按设计稿** | AI 匹配在 50/50 分屏中，对当前查看的岗位进行匹配 | 与 50/50 分屏绑定 |
| **B. 保持现状** | AI 匹配在简历工具栏中 | 无改动 |

### 分歧点

| | 我的方案 | 顾问方案 |
|--|---------|---------|
| 首选 | A. 按设计稿（与 50/50 分屏整合） | 未明确提及 |

### 待决策项
- [ ] AI 匹配功能是否与 50/50 分屏绑定？
- [ ] AI 匹配的交互流程是否需要调整？

### 我的解答
**与 50/50 分屏绑定，按设计稿组织。**

原因：

- 用户未单独提出要保留当前 AI 匹配位置
- 该部分属于“未提到部分”，因此应按设计稿走

结论：

- 当进入 50/50 分屏查看具体岗位时，AI 匹配围绕“当前岗位 + 当前简历”展开
- 交互流程应随该上下文调整，而不是继续保持当前松散位置
