## 疑问1：PDF 解析方案（pdf-parse 已失效）

### 背景
系统需要支持用户上传 PDF 简历，提取文本内容后转为 Markdown 存储。当前环境为 Node.js v24。

### 目标
在 Node.js v24 环境下，可靠地将 PDF 文件提取为纯文本。

### 问题
- `pdf-parse`（npm 最流行的 PDF 解析库）在 Node.js v24 上已失效
- 报错：`PDFParse` class constructor 无法正常调用
- 原因：Node.js v24 改变了 CommonJS 模块解析行为，`pdf-parse` 内部依赖的 `pdf.js` 版本过旧

### 可选方案

| 方案 | 优点 | 缺点 | 风险 |
|------|------|------|------|
| **A. pdfjs-dist**（Mozilla 官方） | 维护活跃、Node.js 兼容性好、文档完善、纯 JS | 包体积较大（~2MB）、API 较底层、纯文本输出无结构 | 低 |
| **B. Microsoft MarkItDown**（Python） | 40K+ stars、CJK 支持好、轻量级、支持 OCR 图片描述 | Python 依赖、与项目 JS 技术栈不匹配 | 中 |
| **C. Marker**（ML-based） | 33K+ stars、高精度保留排版 | 依赖 Python + ML 模型 + GPU、不适合浏览器扩展 | 极高 |
| **D. 跳过 PDF 上传** | 零开发成本 | 用户无法上传 PDF 简历 | 产品层面 |

> **研究补充**：GitHub 研究显示，PDF→MD 是格式转换中**格式丢失最严重**的方向。标题层级、表格结构、分栏布局均可能丢失。这是 PDF 格式本身的限制（面向显示而非面向语义）。对于简历场景，纯文本 + 人工微调是最务实的方案。

### 分歧点
| | 我的方案 | 顾问方案 |
|--|---------|---------|
| 首选 | A. pdfjs-dist | A. pdfjs-dist |

> 两者一致推荐 pdfjs-dist，但均未实际测试验证。

### 待决策项
- [ ] 是否采用 pdfjs-dist？（需要实际测试验证）
- [ ] 是否需要限制 PDF 文件大小？（测试中发现 66MB PDF 导致内存问题，但用户说明实际简历 PDF 很小）

### 测试状态

## 疑问2：PDF 导出方案

### 背景
用户在编辑器中完成简历编辑后，需要能导出为 PDF 格式用于投递。当前环境为 Chrome 浏览器扩展（popup.html），后端为 Node.js（controller/server.js）。

### 目标
将 Markdown 格式的简历内容导出为排版正确的 PDF 文件。

### 可选方案

| 方案 | 环境 | 优点 | 缺点 | 风险 |
|------|------|------|------|------|
| **A. @media print + window.print()** | 前端 | 零依赖、极轻量（~40行CSS） | 需要用户手动操作、跨浏览器不一致、排版控制有限 | 低 |
| **B. 后端 Puppeteer API** | 后端 | 高保真矢量渲染、完全自动化、行业标准（Resume.io 等均采用） | 需要安装 Chromium（~300-500MB）、后端增加 API 端点 | 中 |
| **C. html2pdf.js** | 前端 | 纯前端、API 简单 | **输出为栅格化图片**（文字不可选中/搜索）、大文件体积大、打印可能模糊 | 中-高 |
| **D. md-to-pdf**（Puppeteer封装） | 后端 | 高保真、支持自定义 CSS | 依赖 Chromium、有已知安全漏洞（front-matter JS 注入） | 中 |

> **研究关键发现**：
> - **html2pdf.js 的致命缺陷**：其原理是 html2canvas（截图）+ jsPDF（拼合），输出的是**栅格化图片式 PDF**，文字不可选中、不可搜索、打印效果可能模糊。产品调研评级仅 3/5，不推荐用于专业简历。
> - **@media print 评级仅 2/5**：跨浏览器不一致（Chrome/Firefox/Edge 输出不同），无法控制页眉页脚，用户体验差。
> - **Puppeteer 评级 5/5**：生产级 PDF 生成的首选方案，Resume.io、Canva 等行业标杆均采用服务端 HTML-to-PDF。
> - **可行架构**：前端发送 Markdown → 后端用 marked 渲染 HTML → 后端 Puppeteer 生成 PDF → 返回 Blob。

### 分歧点
| | 我的方案 | 顾问方案 |
|--|---------|---------|
| 首选 | A. @media print（纯前端零依赖） | B. 后端 Puppeteer（高保真） |
| 理由 | 零依赖、符合"最小改动原则"、Chrome 扩展环境最稳定 | @media print 跨浏览器不一致、不够自动化 |

### 修正后的我的分析
基于研究数据，我**修正建议为方案 B（后端 Puppeteer API）**：
- @media print 在产品调研中仅获 2/5 评分，跨浏览器不一致是硬伤
- 后端已有 Node.js 环境，增加一个 PDF 导出 API 端点的改动可控
- 行业标准做法就是后端渲染 + Puppeteer
- 前端只需发送请求、接收 Blob、触发下载，用户体验一致

### 待决策项
- [ ] 采用 @media print（纯前端最简）还是后端 Puppeteer API（行业标准）？
- [ ] 如果采用 Puppeteer，是否需要考虑 Chromium 安装的部署问题？

### 测试状态
- [ ] @media print 方案在 Chrome 扩展 popup 中的实际效果验证（待执行）
- [ ] 后端 Puppeteer API 原型验证（待执行）

## 疑问3：Word 导出方案

### 背景
用户需要将简历导出为 Word 格式，方便在 Word/WPS 中继续编辑或投递。

### 目标
将 Markdown 简历内容导出为 Word/WPS 可正常打开的文档，中文无乱码。

### 可选方案

| 方案 | 环境 | 优点 | 缺点 | 风险 |
|------|------|------|------|------|
| **A. `docx` npm 包**（声明式 API） | 前端+后端 | 生成**真正的 .docx**、周下载 100万+、**浏览器+Node.js 双环境**、中文支持 `font: { eastAsia: "微软雅黑" }` | 需要学习 API、复杂布局实现较繁琐 | 低 |
| **B. HTML 包裹为 .doc**（Blob 下载） | 前端 | 零依赖、~50行代码 | 不是真正的 .docx、Word 打开会提示格式兼容性、WPS/LibreOffice 兼容性差 | 中 |
| **C. @m2d/md2docx**（Unified 生态） | 后端 | 基于 AST 的精确转换、支持表格/引用/脚注/代码块 | 相对较新、社区较小 | 中 |
| **D. docxtemplater**（模板驱动） | 后端 | 在 Word 中设置好模板后注入数据、中文天然支持 | 需要维护 .docx 模板文件 | 低 |

> **研究关键发现**：
> - **`docx` npm 包是 JS Word 生成的最佳方案**（5/5 推荐度），同时支持浏览器和 Node.js 环境。
> - **HTML 包裹 .doc 是 hack 方案**（2/5 推荐度），打开时 Word 会提示格式兼容性警告，WPS/LibreOffice 中兼容性差。
> - **docxtemplater**（4/5 推荐度）适合模板驱动场景，可在 Word 中直接设置中文字体。
> - **@m2d/md2docx** 基于 Unified.js（remark/rehype）生态，适合需要 AST 操作的复杂转换。

### 分歧点
| | 我的方案（原） | 顾问方案 | 我修正后的建议 |
|--|---------|---------|---------|
| 首选 | B. HTML 包裹 .doc | html-to-docx | **A. `docx` npm 包** |

> 基于研究数据，我**修正建议**：`docx` npm 包比 HTML 包裹 .doc 优秀得多，且同时支持前端和后端，是最佳选择。

### 修正后的我的分析
- `docx` npm 包周下载量 100万+，积极维护，同时支持浏览器和 Node.js
- 声明式 API 可以精确控制字体（含中文 eastAsia 字体）、表格、列表等
- 比 HTML 包裹 .doc 方案更专业，输出真正的 .docx 格式
- 前端可通过 CDN 引入或在 content_scripts 中打包使用

### 待决策项
- [ ] 采用 `docx` npm 包（真正的 .docx）还是 HTML 包裹 .doc（最简方案）？

### 测试状态
- [ ] `docx` npm 包在浏览器环境中的可用性测试（待执行）
- [ ] `docx` npm 包生成含中文的 .docx 在 Word/WPS 中的打开效果（待执行）

## 疑问4：Markdown 中的 Base64 图片处理

### 背景
DOCX→MD 转换测试中，mammoth 将 DOCX 中的图片提取为 base64 编码的 data URI 内嵌在 Markdown 中（如 `![image](data:image/png;base64,...)`）。

### 目标
确定 Markdown 存储中图片的最佳处理策略。

### 问题
- Base64 字符串极长（一张图片可达数KB到数十KB），导致 content_md 字段膨胀
- 用户在 textarea 中编辑时，base64 字符串会干扰阅读和编辑
- 回环测试（MD→HTML→MD）中，base64 图片始终保留，确认不会丢失

### 可选方案

| 方案 | 优点 | 缺点 | 风险 |
|------|------|------|------|
| **A. 保持原样**（base64 内嵌） | 零处理、图片不丢失 | content_md 膨胀、编辑体验差 | 低（功能层面） |
| **B. 上传为文件引用** | content_md 瘦身、编辑体验好 | 需要文件存储服务、增加复杂度 | 中 |
| **C. 剥离图片，仅保留文本** | content_md 最小 | 图片信息丢失 | 高（产品层面） |

### 分歧点
无分歧。顾问和我的观点一致：实际简历中图片极少（通常只有一张头像照），base64 膨胀问题在简历场景下不严重。

### 我的建议
**方案 A（保持原样）**。理由：
- 简历中通常只有一张头像照，base64 不会造成严重的存储问题
- 如果未来确实需要优化，可以在 M10+ 迭代中升级为文件引用
- 当前保持简单，避免引入不必要的复杂度

### 待决策项
- [ ] 是否接受 base64 内嵌方案？

## 疑问5：Markdown 列表符号不一致

### 背景
DOCX→MD 转换测试中，turndown 将 DOCX 的列表项转换为 `- ` 开头的 Markdown 列表。但 MD→HTML→MD 回环测试中，marked 渲染后再用 turndown 转回时，列表符号变为 `* `。

### 目标
确保 Markdown 在多次转换后保持一致。

### 问题
- `- item` 和 `* item` 在 Markdown 中语义相同，渲染结果一致
- 但多次转换会导致 git diff 中出现大量无意义的符号变化
- 回环测试显示 52.8% 完全匹配，0 内容丢失，差异主要是这类格式变化

### 可选方案

| 方案 | 优点 | 缺点 |
|------|------|------|
| **A. 不处理** | 零代码、不影响功能 | 回环时产生格式漂移 |
| **B. turndown 配置统一** | 源头统一 | 需要配置 turndown 的 bulletListMarker 选项 |

### 分歧点
无分歧。

### 我的建议
**方案 B**。在 turndown 初始化时配置：
```js
new TurndownService({ bulletListMarker: '-' });
```
一行配置即可解决，无副作用。

### 待决策项
- [ ] 是否在 turndown 配置中统一列表符号？

## 疑问6：PDF 文本结构丢失

### 背景
PDF 本质上是页面布局描述（文字位置、字体、大小），而非语义化文档。从 PDF 提取纯文本时，标题、段落、列表等结构信息可能丢失。

### 目标
尽可能从 PDF 中恢复简历的语义结构。

### 问题
- pdfjs-dist 提取的是按渲染顺序排列的文本块，不包含 h1/h2/ul 等语义标签
- 不同 PDF 生成工具（Word 导出、WPS 导出、在线生成）的文本提取结果差异很大
- 简历中的分栏布局（如左右两栏）提取后可能变成交错文本

### 可选方案

| 方案 | 优点 | 缺点 |
|------|------|------|
| **A. 纯文本 + AI 结构化** | 用 LLM 恢复标题/段落结构 | 依赖 AI、增加延迟和成本 |
| **B. 纯文本直接存储** | 简单快速 | 结构信息丢失、用户需要手动调整 |
| **C. 使用 Marker/Docling 等智能解析** | 高精度结构恢复 | 依赖 Python、不适合 Chrome 扩展 |

### 分歧点
| | 我的方案 | 顾问方案 |
|--|---------|---------|
| 首选 | B. 纯文本直接存储（M9阶段） | A. AI 结构化（利用 M8 已有的 LLM 能力） |
| 理由 | M9 应先跑通基础流程，结构化可后续迭代 | M8 的 LLM 基础设施已就绪，可以复用 |

### 我的分析
- 简历 PDF 的结构通常较简单（标题、段落、列表），纯文本 + 人工微调足够
- AI 结构化作为可选增强功能，不应阻塞 M9 基础交付
- 可以在导出时提供"AI 优化"按钮，而非自动处理

### 待决策项
- [ ] M9 阶段是否接受纯文本方案，AI 结构化推迟到后续迭代？

## 疑问8：PDF 导出中文字体支持

### 背景
使用 @media print 或任何前端 PDF 生成方案时，需要确保中文字符正确渲染。

### 目标
导出的 PDF 中中文显示正常，无乱码、无方框替代。

### 问题
- 浏览器默认字体栈中可能不包含中文字体
- Linux 系统中文字体支持不如 Windows/macOS 完善
- 如果使用 md-to-pdf（Puppeteer），需要在 CSS 中显式指定中文字体

### 可选方案

| 方案 | 说明 |
|------|------|
| **A. 通用字体栈** | CSS 中指定 `font-family: 'PingFang SC', 'Microsoft YaHei', 'Noto Sans CJK SC', sans-serif` |
| **B. 内嵌 Web 字体** | 通过 Google Fonts 或本地加载 Noto Sans CJK |
| **C. 系统字体检测** | JS 检测可用中文字体后动态设置 |

### 分歧点
无分歧。

### 我的建议
**方案 A（通用字体栈）**。理由：
- @media print 方案中，浏览器使用系统字体渲染，不需要内嵌
- 通用字体栈覆盖了三大操作系统（macOS: PingFang, Windows: 微软雅黑, Linux: Noto）
- 零额外依赖

### 待决策项

## 疑问9：MD→HTML 渲染库选型（marked vs markdown-it）

### 背景
当前 M9 工作包中使用 `marked` 库进行 Markdown→HTML 渲染。但研究显示 `markdown-it` 在社区中被更广泛推荐。

### 目标
选择最适合的 Markdown→HTML 渲染库。

### 可选方案

| 方案 | npm 下载量 | 优点 | 缺点 |
|------|-----------|------|------|
| **A. marked** | 历史最高（legacy 项目多） | 已集成、回环测试已验证 | 规范兼容较松（CommonMark-ish）、扩展性有限 |
| **B. markdown-it** | 2024-2025 社区首选 | CommonMark 兼容性好、插件生态完善、安全记录较好 | 需要替换现有集成代码 |

> **研究关键发现**：
> - 多个技术博客（如 macwright.com）明确建议"Don't use marked"
> - markdown-it 是 2024-2025 年新项目的社区首选
> - markdown-it 支持中文排版插件

### 分歧点
无分歧（此问题之前未被讨论）。

### 我的建议
**方案 A（继续用 marked）**。理由：
- M9 的 Markdown 渲染场景简单（简历内容），不需要复杂的插件生态
- 已通过回环测试验证，替换的风险 > 收益
- 如果未来需要更丰富的渲染能力，可以在后续迭代中替换

### 待决策项
- [ ] 是否替换为 markdown-it？（建议：M9 阶段不替换）

## 研究结论摘要

### PDF→MD（上传解析）
PDF 是"面向显示"的格式，**格式丢失不可避免**。对于简历场景，纯文本 + 人工微调最务实。

### MD→PDF（导出）
- 纯前端方案（@media print / html2pdf.js）质量不足以用于专业简历
- **行业标准是后端 Puppeteer**（Resume.io、Canva 均采用）
- 推荐架构：前端发送 MD → 后端渲染 HTML → Puppeteer 生成 PDF → 返回 Blob

### MD→DOCX（导出）
- HTML 包裹 .doc 是 hack 方案（2/5 评分）
- **`docx` npm 包**是最佳 JS 方案（5/5 评分），同时支持浏览器和 Node.js
- 中文通过 `font: { eastAsia: "微软雅黑" }` 原生支持

### 完整推荐技术链
```
上传解析：
  DOCX → mammoth → HTML → turndown → Markdown（已验证✅）
  PDF  → pdfjs-dist → 纯文本 → Markdown（待测试）

导出：
  Markdown → `docx` npm → .docx（真正的 DOCX）⭐推荐
  Markdown → 后端 Puppeteer → .pdf（矢量高保真）⭐推荐
  Markdown → Blob 下载 → .md（零成本，已验证✅）
  Markdown → 内联 CSS HTML → .html（零成本，已验证✅）
```

## 我的补充解答（围绕“高保真 + 控体积 + 可读性/性能优先”）

下面这组结论不是重复研究摘要，而是针对“另外两个人没想通的点”给出的**决策准则**。核心原则只有一条：

> **不要追求三格式“字节级完美对等”，要追求“语义完整 + 可读版式稳定 + 输出体积可控”。**

这三种格式的物理特性不同：

- **Markdown**：最适合编辑、最轻量、最适合作为内部事实源
- **DOCX**：最适合继续编辑、保留结构、兼容 Word/WPS
- **PDF**：最适合最终投递、版式冻结、高保真展示

因此，M9 最稳的产品策略应该是：

### 1. 以 Markdown 作为唯一编辑事实源（SSOT）

建议明确写死：

- 编辑态只认 `content_md`
- HTML / DOCX / PDF 都是**派生输出**
- 不反向把 PDF/DOCX 当作长期事实源保存

原因：

- Markdown 最轻，最适合做 diff、存储、LLM 处理和 textarea 编辑
- 如果把三种格式都当事实源，后续必然出现“哪份是最新版本”的同步问题
- 你要的是“完美转换体验”，不是“三套主存”

### 2. “完美转换”的正确目标定义

我建议在文档里明确把“完美转换”定义为：

1. **标题、段落、列表、强调、链接、表格等语义不丢**
2. **中文可读、分页稳定、排版不崩**
3. **导出的 PDF/DOCX 能直接用于投递或二次编辑**
4. **文件体积不过度膨胀**

不建议把“像素级完全一致”作为目标，因为：

- Markdown 本身不表达精确版面
- DOCX 和 PDF 的渲染引擎不同
- 为追求像素级一致，往往要引入大量内联样式、base64 资源和复杂布局，直接伤害体积与性能

### 3. PDF 只做“最终版导出”，不要参与回环

这是最关键的一条。

建议补一句硬规则：

- **PDF 是输出终点，不参与回环编辑链。**

也就是说：

- `MD → PDF` 要高保真
- `PDF → MD` 只要求“文本和核心结构尽量恢复”，不要求回到原版面

这样可以把问题拆开：

- 上传 PDF 简历时：接受结构损失，优先文本提取成功率
- 导出 PDF 简历时：优先排版质量和打印质量

这正好解释为什么：

- **PDF 解析** 用 `pdfjs-dist` 就够了
- **PDF 导出** 应该用后端 Puppeteer

两者不是同一个质量目标，不要强求对称。

### 4. DOCX 应该承担“可编辑高保真”的角色

在三格式里，真正应该承接“完美转换”责任的是 DOCX，而不是 PDF。

我的建议：

- `Markdown → DOCX` 作为**高保真编辑导出主链**
- `Markdown → PDF` 作为**高保真投递导出主链**
- `PDF → Markdown` 作为**降级导入链**

这意味着：

- `docx` npm 包的优先级其实和 Puppeteer 一样高
- 如果要节约空间和复杂度，优先把精力花在：
  - 标题层级映射
  - 列表/表格
  - 中文字体
  - 段落间距
  - 页边距和分页控制

而不是追求炫技式复杂样式

### 5. 体积控制的决策建议

你提到“能节约一点空间是一点”，我认同，但不能牺牲关键目标。建议按下面优先级控制体积：

#### 必做

- **Markdown 存储中不内联无必要样式**
- **导出 HTML 使用共享 CSS 模板，不内联大段重复样式**
- **Puppeteer 渲染 PDF 时禁用不必要图片/动画/阴影**
- **DOCX 中只用必要字体和段落样式，不堆过多 run 级别样式**

#### 可以接受

- 一张头像图的 base64 或资源引用
- 少量表格、粗体、列表带来的体积增长

#### 不建议

- 为了“所见即所得”把整个 HTML/CSS 全量内联到 DOCX/PDF 生成链
- 用 html2canvas 一类截图方案生成 PDF
- 在 Markdown 中长期保存超长 base64 图片串，尤其多图场景

### 6. 关于 Base64 图片：当前可接受，但应写清升级阈值

我同意文档里“简历场景图片少，当前可以保留 base64”的判断。  
但建议补一句阈值规则：

- **单份简历若图片总量超过 1MB 或图片数量超过 2 张，就应转入文件引用方案候选。**

这样既不提前复杂化，也避免未来内容膨胀没有处理标准。

### 7. 关于中文和版式：优先“稳定字体栈”，不要过度定制

建议把中文导出的优先策略写成：

- PDF：固定字体栈 + 固定页边距 + 控制标题/段落/列表 spacing
- DOCX：固定正文样式、标题样式、列表样式、表格样式

不要在 M9 追求：

- 多字体混排
- 花哨主题
- 复杂色块和多栏排版

简历的目标是：

- 可读
- 可编辑
- 可打印

不是做设计作品集。

### 8. 对疑问6（PDF 结构丢失）的最终建议

我支持你当前倾向的：

- **M9 阶段接受“纯文本 + 人工微调”**
- AI 结构化放到后续增强能力

但建议补一句更精确的话：

- **如果 M8 的 LLM 能力已经稳定，可把“AI 结构恢复”做成可选按钮，不默认执行，不阻塞上传链路。**

这样可以兼顾：

- 基础链路稳定
- 后续可读性增强
- 成本和延迟可控

### 9. 对疑问9（marked vs markdown-it）的最终建议

我同意 M9 不替换，继续用 `marked`。  
原因不是它“最好”，而是当前目标下：

- 你需要的是稳定、轻量、已验证
- 不是扩展丰富 Markdown 语法

建议把结论写成：

- **M9 保持 `marked`，把“切换到 markdown-it”标记为未来仅在出现明确渲染缺陷时才触发的优化项。**

这样可以避免无意义替换。

### 10. 最终推荐的产品级决策

如果要一句话定策略，我建议写成：

> **M9 以 Markdown 为唯一事实源，DOCX 负责高保真可编辑导出，PDF 负责高保真最终投递导出；解析链优先稳定与轻量，导出链优先可读性、打印质量与文件体积控制。**

这个定义能同时满足：

- 三格式转换体验尽量好
- 空间尽量省
- 性能和维护成本可控
- 不陷入“每个方向都追求完全对称”的过度设计

## 顾问确认与技术补充（2026-03-26）

> 顾问结论：10 条决策准则逻辑严密，技术方案**完全可行**。以下为顾问补充的 4 个关键防范点。

### 防范1：Puppeteer 中文字体缺失风险

即便 CSS 中写了 `font-family: 'Microsoft YaHei', 'Noto Sans CJK SC'`，如果后端部署在 Linux/Docker 中，系统默认没有这些字体，导出的 PDF 中文会变成方框。

**对策**：在部署脚本/Dockerfile 中显式安装 `fonts-noto-cjk`。

### 防范2：Puppeteer 并发与内存

Puppeteer 启动浏览器开销大，建议：
- 使用 `puppeteer-core` 配合已安装的 Chrome
- 使用浏览器实例池（如 `generic-pool`）或复用 Browser 实例
- 每次请求只开新 Page，避免内存溢出

### 防范3：Chrome 扩展 MV3 限制

在 MV3 架构下，后台 Service Worker 无法直接调用 DOM API 或触发下载。`docx` 生成 Blob 并触发下载的逻辑**必须放在 Popup 页面或 Content Script 中执行**，不能放在 Background 脚本中。

### 防范4：pdfjs-dist 版本/Worker 陷阱

Node.js v24 下 ESM vs CJS 解析严格。建议：
- 使用 `pdfjs-dist/legacy/build/pdf.js`
- 确保 `package.json` 的 `type` 配置与引入方式匹配
- Node 端需禁用 Worker（`GlobalWorkerOptions.workerSrc = ''`）

### 顾问补充建议

- **Base64 图片优化**：利用 Canvas API 在前端将上传头像压缩到 200KB 以内再转 Base64，可无限期推迟"引入文件存储"的复杂性
- **Puppeteer 并发测试**：验证后端连续生成 5-10 份 PDF 时的内存占用

**状态：顾问已确认，无分歧**
