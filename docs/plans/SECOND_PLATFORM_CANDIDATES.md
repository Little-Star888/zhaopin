## 2. 逐平台评估

### 2.1 猎聘

#### 技术架构

- **渲染方式**：前后端分离，CSR（客户端渲染）为主，核心职位数据通过 AJAX/XHR 异步请求动态加载
- **API 端点**：`https://api-c.liepin.com/api/com.liepin.searchfront4c.pc-search-job`（已知公开端点）
- **数据格式**：JSON 响应，包含职位列表数组和分页信息
- **URL 结构**：`liepin.com/job/{jobId}.shtml`，职位 ID 嵌入 URL 路径

#### 反爬机制

| 反爬层级 | 具体措施 | PoC 应对难度 |
|---------|---------|-------------|
| IP 层面 | 频率限制、IP 封锁 | 中（Chrome 扩展使用用户真实 IP，频率可控） |
| 请求层面 | User-Agent 检测、Referer 校验 | 低（Chrome 扩展天然携带正确 UA 和 Referer） |
| 身份验证 | Cookie/Token 机制，需登录态 | 低（扩展可检测 `chrome.cookies`，复用用户已登录会话） |
| 验证码 | 触发频率阈值后弹出图形/滑块验证码 | 中（需手动辅助，与 Boss 现有模式一致） |
| 数据加密 | 部分接口参数存在签名/加密机制 | 高（直接 API 调用需要逆向 JS 签名算法，但 Chrome 扩展方案可绕过） |
| 行为检测 | 浏览器指纹、鼠标轨迹/滚动行为检测 | 低（Chrome 扩展运行在真实浏览器环境，天然通过） |

## 3. 候选排序

| 排名 | 平台 | 综合得分 | 推荐理由 |
|------|------|---------|---------|
| **1** | **猎聘** | **3.95** | 数据可获取性好（JSON API 已知、Chrome 扩展可复用登录态）；模型相似度高（8/12 标准字段低难度映射）；商业价值最高（中高端市场领先，与 Boss 形成人群互补）；反爬难度中等但 Chrome 扩展方案可规避；PRD 已将其列为 PoC 首选 |
| **2** | **智联招聘** | **3.25** | 反爬难度最低（传统平台，防护层级较少）；模型相似度中高；但数据可获取性待实际验证（API 细节未知）；商业价值中等（与 Boss 重叠度高，互补性弱）；适合作为第三平台接入 |
| **3** | **51job** | **2.95** | 反爬体系最复杂（设备指纹+行为时序+动态鉴权+流量特征四维防护）；公开逆向文档稀缺；新版域名接口结构待确认；商业价值不错但与 Boss 重叠度高；建议放在最后，待前两个平台 PoC 成功后再评估 |

### 排序结论

**推荐 PoC 顺序：猎聘 > 智联招聘 > 51job**

与 PRD 建议顺序的差异说明：
- PRD 建议"猎聘 > 51job > 智联招聘"
- 本评估建议将智联招聘提升到第二位，原因：智联招聘的反爬难度显著低于 51job（评分 4 vs 2），更适合作为 Adapter/Executor 抽象验证的第二步
- 51job 的反爬复杂度（四维防护）可能导致 PoC 阶段消耗过多精力在反爬对抗上，偏离"验证 adapter/executor 抽象层"的核心目标

## 4. 风险评估

| 风险编号 | 风险模块 | 潜在问题 | 发生概率 | 影响范围 | 应对方案 |
|---------|---------|---------|---------|---------|---------|
| R1 | 数据可获取性 | 猎聘 API 接口参数存在签名加密，直接 HTTP 调用被拦截 | 中 | 猎聘 Executor 实现受阻 | 采用 Chrome 扩展方案，在真实浏览器环境中执行搜索，天然绕过签名验证；不尝试直接调用 API |
| R2 | 数据可获取性 | 各平台接口结构在 PoC 前无法完全确认，字段映射可能需大幅调整 | 高 | Adapter 映射逻辑返工 | 严格遵循 UNIFIED_JOB_MODEL_V2 的"待验证"标记机制；PoC 第一阶段仅做列表抓取，确认字段后再做详情抓取；扩展字段先存入 `platformMetadata`，不预设映射 |
| R3 | 反爬升级 | 各平台持续升级反爬策略，Chrome 扩展方案可能被检测（如 WebDriver 标识检测） | 中 | 所有平台 Executor 受影响 | 扩展不使用自动化驱动（Selenium/Playwright），而是利用 Content Script 注入 + 用户手动操作触发采集；避免 `navigator.webdriver` 等自动化标识暴露 |
| R4 | 登录态维持 | 各平台 Cookie 有效期不同，登录态过期后需要用户重新登录 | 高 | 采集任务中断 | 参考 `chrome.cookies.onChanged` 监听登录态变化（PRD 6.1）；提供"登录状态检测"按钮；任务中断后支持断点续传 |
| R5 | 模型差异 | 三个平台的薪资展示格式差异大（猎聘规范、51job 有面议、智联有面议），统一标准化困难 | 中 | `salary` 字段标准化质量 | `salary` 字段设计为"原始文本"（UNIFIED_JOB_MODEL_V2 已确认），不做结构化解析；后续 enrichment 阶段再统一为 `salaryMin/salaryMax` |
| R6 | 城市编码 | Boss 使用天气编码格式（如 `101010100`），其他平台使用不同编码体系 | 高 | 跨平台城市筛选失效 | Controller 统一使用城市名称（如"北京"），由各平台 Executor 内部维护城市名称到平台编码的映射表（BOSS_SPECIFIC_BOUNDARY R3 已识别） |
| R7 | 合规风险 | 大规模采集招聘数据可能违反《反不正当竞争法》《数据安全法》《个人信息保护法》 | 低 | 项目法律风险 | 仅采集公开职位信息，不采集个人联系方式；控制采集频率，不进行大规模批量抓取；猎聘设有 LPSRC 安全应急响应中心，需注意合规边界 |

## 5. PoC 策略（仅文档层面）

### 5.1 总体策略

PoC 的核心目标不是"成功抓取数据"，而是**验证 Adapter/Executor 抽象层是否能支撑多平台接入**。因此策略应聚焦于：

1. 用最小成本拿到一个平台的真实数据
2. 用真实数据验证 UNIFIED_JOB_MODEL_V2 的字段映射假设
3. 验证 Controller 层的多平台编排逻辑（去重、持久化、enrichment）

### 5.2 PoC 步骤

| 步骤 | 内容 | 验证方式 | 预期产出 |
|------|------|---------|---------|
| **Step 1: 接口抓包分析** | 使用 Chrome DevTools 对猎聘搜索列表页进行 F12 抓包，记录：(1) XHR 请求 URL 和参数；(2) 响应 JSON 结构；(3) 字段名称和含义；(4) 分页参数；(5) 登录态要求 | 手动在浏览器中操作，截图记录 Network 面板内容 | 《猎聘接口抓包记录》，包含请求/响应示例 |
| **Step 2: 字段映射验证** | 将抓包获取的猎聘字段与 UNIFIED_JOB_MODEL_V2 标准字段逐一对照，确认：(1) 哪些标准字段可直接映射；(2) 哪些需要组合或转换；(3) 哪些猎聘特有字段需存入 platformMetadata；(4) 薪资/经验/学历的原始格式 | 填写字段映射表，与 UNIFIED_JOB_MODEL_V2 3.1 节对比 | 《猎聘字段映射表》v1，标注每个字段的映射难度和来源 |
| **Step 3: 列表页最小 PoC** | 通过 Chrome 扩展 Content Script 在猎聘搜索页面注入脚本，执行一次搜索并解析列表页返回的职位数据，产出 UnifiedJob 格式的 JSON 数组 | 在开发环境中执行扩展，检查输出 JSON 是否符合 Adapter 输出接口定义（UNIFIED_JOB_MODEL_V2 6.3 节） | 至少 5 条 UnifiedJob 格式的职位数据样本 |
| **Step 4: 详情页最小 PoC** | 对 Step 3 获取的职位，进入详情页抓取完整职位描述、福利标签等扩展字段 | 检查 `description`、`platformMetadata` 是否正确填充 | 至少 2 条含详情数据的完整 UnifiedJob 样本 |
| **Step 5: 入库验证** | 将 PoC 获取的 UnifiedJob 数据通过现有 Controller 流程写入 `scraped_jobs` 表，验证：(1) `UNIQUE(platform, platformJobId)` 约束生效；(2) `platform = 'liepin'` 正确；(3) 查询 API 可正常返回猎聘数据 | 查询 `scraped_jobs` 表，验证数据完整性和查询可用性 | 入库成功的截图/日志，查询结果确认 |

### 5.3 PoC 成功标准

| 标准 | 具体要求 |
|------|---------|
| 列表抓取 | 能稳定获取猎聘搜索结果列表，至少包含 title/company/location/salary |
| 详情抓取 | 能获取职位描述和扩展字段 |
| 模型映射 | Adapter 输出符合 UnifiedJob 接口定义，Controller 可正常处理 |
| 入库成功 | 数据写入 `scraped_jobs` 表，`platform = 'liepin'`，去重约束生效 |
| 不影响 Boss | Boss 原有采集流程不受影响，现有数据完整 |

### 5.4 PoC 失败退出条件

| 退出条件 | 处理方式 |
|---------|---------|
| 猎聘列表接口签名加密导致 Chrome 扩展也无法获取数据 | 评估是否需要 JS 逆向，或切换到智联招聘作为第一 PoC 目标 |
| 字段映射发现标准字段大量缺失（> 5 个标准字段无法获取） | 重新评估 UNIFIED_JOB_MODEL_V2 的字段设计，考虑降低必填字段要求 |
| 登录态检测失败，无法稳定获取用户 Cookie | 评估 `chrome.cookies` API 的兼容性，或采用手动导入 Cookie 方案 |

## 附录 B：评分方法说明

- **评分范围**：1（极差）到 5（极好）
- **加权计算**：综合得分 = 各维度评分 x 权重 之和
- **PoC 优先原则**：数据可获取性和模型相似度合计权重 55%，是 PoC 阶段最关键的两个维度
- **不考察市场占有率**：商业价值维度侧重"与 Boss 的互补性"和"目标用户使用价值"，而非纯市场号码
